# 工作流降级报告: 从 DSL v0.4.0 到 v0.3.0

## 📋 **1. 降级概述**

### **目标**
将使用 DSL v0.4.0 规范创建的 `香水短视频分析工作流_v3_validated.yml` 文件，降级为兼容 Dify 系统支持的 **DSL v0.3.0** 版本，解决因版本不匹配导致的导入失败问题。

### **核心变更**
v0.4.0 的 DSL 结构更为精简和数据驱动，而 v0.3.0 的结构包含了更多与前端 UI 展示相关的冗余信息。本次降级的核心是将 v0.4.0 的精简结构 **还原** 为 v0.3.0 更为详细、层级更复杂的结构。

- **源文件**: `香水短视频分析工作流_v3_validated.yml` (v0.4.0)
- **目标文件**: `香水短视频分析工作流_v0.3.0_compatible.yml` (v0.3.0)

---

## 🔧 **2. 降级调整内容清单**

以下是本次降级操作中具体的调整项目：

### **变更 #1: 修改文件版本号**
- **操作**: 将 `version` 字段从 `0.4.0` 修改为 `0.3.0`。
- **原因**: 明确声明文件符合 v0.3.0 的 DSL 规范。
- **v0.4.0**: `version: 0.4.0`
- **v0.3.0**: `version: 0.3.0`

---

### **变更 #2: 调整 `nodes` 节点结构**
- **操作**: 为 `graph.nodes` 中的每个节点增加用于 UI 展示的顶层字段，还原 v0.3.0 的复杂结构。
- **原因**: v0.3.0 的解析器需要这些额外的（即使是冗余的）UI 字段才能正确识别节点。
- **v0.4.0 结构 (精简)**:
  ```yaml
  nodes:
    - id: start-node
      data:
        title: "开始"
        type: start
        # ... 其他核心配置
  ```
- **v0.3.0 结构 (复杂)**:
  ```yaml
  nodes:
    - id: start-node
      data: # 核心配置保留在 data 内部
        title: "开始"
        type: start
        # ...
      # 新增的 UI 字段
      height: 116
      position: { x: 30, y: 282 }
      positionAbsolute: { x: 30, y: 282 }
      selected: false
      sourcePosition: right
      targetPosition: left
      type: custom # v0.3.0 需要一个顶层的 type 字段
      width: 244
  ```

---

### **变更 #3: 调整 `edges` 连接结构**
- **操作**: 为 `graph.edges` 中的每个连接增加 `data` 对象和 UI 相关字段。
- **原因**: v0.3.0 的连接定义需要一个 `data` 对象来包裹类型信息，并需要额外的 UI 字段。
- **v0.4.0 结构 (精简)**:
  ```yaml
  edges:
    - id: edge-start-to-extractor
      source: start-node
      target: doc-extractor-node
  ```
- **v0.3.0 结构 (复杂)**:
  ```yaml
  edges:
    - id: edge-start-to-extractor
      source: start-node
      target: doc-extractor-node
      # 新增的 data 对象和 UI 字段
      data:
        sourceType: start
        targetType: doc-extractor
      sourceHandle: source
      targetHandle: target
      type: custom
      zIndex: 0
  ```

---

### **变更 #4: 降级 `start` 节点文件上传配置**
- **操作**: 将 `start` 节点中 v0.4.0 的 `file` 对象配置，还原为 v0.3.0 的独立字段配置。
- **原因**: v0.3.0 不支持 v0.4.0 中结构化的 `file` 对象。
- **v0.4.0 配置**:
  ```yaml
  variables:
    - type: file
      file:
        max_size: 50
        extensions: [".mp4", ".mov"]
  ```
- **v0.3.0 配置**:
  ```yaml
  variables:
    - type: file
      # 还原为独立字段
      max_length: 50 # max_size -> max_length
      file_types: ["video/mp4", "video/mov"] # extensions -> file_types (MIME types)
  ```

---

### **变更 #5: 还原 `features.file_upload` 配置**
- **操作**: 将 `features.file_upload` 的结构从 v0.4.0 格式还原为 v0.3.0 格式。
- **原因**: v0.3.0 的文件上传特性开关配置结构不同。
- **v0.4.0 配置**:
  ```yaml
  features:
    file_upload:
      image:
        enabled: true
  ```
- **v0.3.0 配置**:
  ```yaml
  features:
    file_upload:
      allowed_file_types: ["video"] # 使用 allowed_file_types 列表
      image:
        enabled: true
  ```

---

## ✅ **3. 结论**

通过执行以上 5 项核心调整，可以将一个 v0.4.0 的工作流文件成功降级到 v0.3.0 版本，从而解决在新版编辑器中创建、但在旧版系统中无法导入的问题。
